## 1. نظرة عامة على المشروع

هذا تطبيق ويب كامل لإدارة أعمال توزيع المواد الغذائية (دهانات). يعمل التطبيق مباشرةً من **ملف HTML واحد** (`index.html`) ويتصل بقاعدة بيانات Supabase السحابية.

### ما يستطيع النظام فعله:
- إدارة **العملاء** وحساباتهم وأرصدتهم
- إنشاء **الفواتير** وتتبع بنودها
- تسجيل **التحصيلات** النقدية وغيرها
- إدارة **المحفظة الإلكترونية** (فودافون كاش)
- تتبع **دفعات الموردين/المصنع**
- حساب **الأرباح** الصافية
- عرض **التقارير** والإحصاءات
- إدارة **المستخدمين** وصلاحياتهم
- **سجل حركات** كامل لمتابعة جميع العمليات

### التقنيات المستخدمة:
| التقنية | الغرض |
|---------|--------|
| HTML/CSS/JavaScript | واجهة المستخدم (ملف واحد) |
| Tailwind CSS (CDN) | التصميم والأنماط |
| Supabase | قاعدة البيانات السحابية |
| Chart.js | الرسوم البيانية |
| jsPDF | تصدير PDF |
| Font Awesome | الأيقونات |

## 3. إعداد قاعدة البيانات على Supabase

### الخطوة 1: إنشاء مشروع Supabase

1. اذهب إلى https://supabase.com
2. سجّل حساباً جديداً (أو ادخل إن كان لديك حساب)
3. اضغط **"New Project"**
4. اختر اسماً للمشروع (مثلاً: `food-distribution`)
5. اختر كلمة مرور قوية لقاعدة البيانات
6. اختر أقرب منطقة جغرافية لك (مثلاً: Europe West)
7. اضغط **"Create new project"** وانتظر دقيقتين حتى يكتمل الإعداد

### الخطوة 2: تشغيل ملف SQL

1. من لوحة تحكم Supabase اذهب إلى: **SQL Editor** (في القائمة الجانبية)
2. اضغط **"New query"**
3. افتح ملف `database/setup.sql` الموجود في هذا المشروع
4. **انسخ كامل محتوى الملف** (Ctrl+A ثم Ctrl+C)
5. الصق المحتوى في محرر SQL في Supabase
6. اضغط **"Run"** (أو Ctrl+Enter)
7. انتظر حتى يظهر رسالة النجاح "Success. No rows returned."

> **ملاحظة مهمة**: الملف آمن للتشغيل أكثر من مرة — إذا نفّذته مرتين لن يحذف أي بيانات موجودة.

### الخطوة 3: الحصول على بيانات الاتصال

1. من لوحة تحكم Supabase اذهب إلى: **Settings → API**
2. انسخ:
   - **Project URL**: مثل `https://xxxxx.supabase.co`
   - **anon public key**: مفتاح طويل يبدأ بـ `eyJhbGciOiJIUzI1NiIs...`

## 4. إعداد المستخدمين في Supabase Auth

> **مهم جداً**: يجب إضافة كل مستخدم في مكانين:
> 1. جدول `users` في قاعدة البيانات (يتم بالـ SQL تلقائياً عند تشغيل setup.sql)
> 2. **Supabase Authentication** (يجب إضافته يدوياً)

### إضافة مستخدم في Supabase Auth:

1. اذهب إلى: **Authentication → Users**
2. اضغط **"Add user" → "Create new user"**
3. أدخل البريد الإلكتروني وكلمة المرور
4. اضغط **"Create user"**

### المستخدمون الافتراضيون (التجريبيون):

بعد تشغيل setup.sql، يجب إضافة هؤلاء المستخدمين في Authentication:

| البريد الإلكتروني | كلمة المرور المقترحة | الدور |
|-------------------|---------------------|-------|
| admin@haramain.com | Admin@2025 | مدير النظام |
| manager@haramain.com | Manager@2025 | مدير عام |
| ahmed@haramain.com | Ahmed@2025 | مبيعات |
| mahmoud@haramain.com | Mahmoud@2025 | مبيعات |
| khaled@haramain.com | Khaled@2025 | حسابات |

> **تعطيل تأكيد البريد الإلكتروني** (مهم):
> اذهب إلى **Authentication → Providers → Email**
> أوقف تشغيل **"Confirm email"** (اجعله OFF)
> وإلا لن يتمكن المستخدمون من تسجيل الدخول حتى يؤكدوا بريدهم

## 5. تشغيل التطبيق

### الطريقة 1: مباشرة من المتصفح (إذا كان Supabase مُعدَّداً مسبقاً)

إذا كانت بيانات Supabase موجودة في ملف `index.html` مسبقاً (في أسطر 440-441) وكان الإعداد مكتملاً:
```
window.SUPABASE_URL  = 'https://xxxxx.supabase.co';
window.SUPABASE_ANON_KEY = 'eyJhbGciOiJIUzI1NiIs...';
```
فقط افتح ملف `index.html` في المتصفح مباشرةً وسيعمل.

### الطريقة 2: إعداد Supabase من داخل التطبيق

1. افتح `index.html` في المتصفح
2. ستظهر صفحة تسجيل الدخول
3. قبل تسجيل الدخول، لا يمكنك الوصول للإعدادات مباشرةً
4. تأكد من أن الملف يحتوي على بيانات Supabase الصحيحة (أسطر 440-441)
5. عدّل هذه الأسطر بقيم مشروع Supabase الخاص بك

### الطريقة 3: رفع الملف على استضافة ويب

يمكنك رفع ملف `index.html` على أي استضافة ويب (مثل Netlify، Vercel، GitHub Pages) وسيعمل مباشرةً.

## 6. شرح صفحات التطبيق

### 📊 لوحة التحكم (Dashboard)
- إحصاءات شاملة للأداء
- يمكن اختيار الفترة الزمنية: اليوم / أسبوع / شهر / ربع / سنة / مخصص
- رسوم بيانية للمبيعات والتحصيلات
- أعلى العملاء والمنتجات مبيعاً
- أداء المندوبين

### 👥 العملاء (Customers)
- قائمة كاملة بجميع العملاء وأرصدتهم
- **الرصيد** = الرصيد الافتتاحي + الفواتير - التحصيلات - المديونيات المعدومة
- إمكانية **البحث** والتصفية حسب المنطقة والحالة والتصنيف
- **كشف حساب** لكل عميل
- تصدير إلى Excel (CSV)
- استيراد من ملف CSV
- **ألوان الرصيد**:
  - 🟢 أخضر: الرصيد صفر أو أقل (لا دين)
  - 🟡 أصفر: من 1 إلى 30 يوم
  - 🟠 برتقالي: من 31 إلى 60 يوم
  - 🔴 أحمر: أكثر من 60 يوم (دين قديم)

### 📦 المنتجات (Products)
- كتالوج المنتجات مع الأسعار
- **سعر الوكيل** هو السعر الذي يُباع به للعميل
- **سعر المصنع** هو التكلفة — يُستخدم لحساب الربح
- تصفية حسب الفئة

### 🧾 الفواتير (Invoices)
- إنشاء وتعديل وإلغاء الفواتير
- كل فاتورة لها رقم تسلسلي فريد (INV-YYYYMMDD-XXXX)
- **بنود الفاتورة**: المنتج، الكمية، السعر، الخصم، الإجمالي
- حالة التوصيل: تم التسليم / قيد التشغيل
- تصدير الفاتورة كـ PDF
- تصدير الفواتير إلى Excel

### 💰 التحصيلات (Collections)
- تسجيل مدفوعات العملاء
- طرق الدفع: نقدي / فودافون كاش / إنستاباي / تحويل بنكي / شيك
- ربط التحصيل برقم الفاتورة (اختياري)

### 📱 فودافون كاش / المحفظة الإلكترونية (Wallet)
- تتبع رصيد المحفظة الإلكترونية
- تسجيل الواردات والمصروفات
- الرصيد يُحسب تراكمياً من الرصيد الافتتاحي

### 🏭 دفعات الموردين (Factory Payments)
- تسجيل المدفوعات للمصنع/الموردين
- يُستخدم لحساب صافي المديونية للمصنع

### 📈 الأرباح (Profits)
- حساب الربح الصافي = صافي المبيعات - تكلفة المصنع
- تفصيل الأرباح حسب العميل والمنتج والمندوب

### 📋 التقارير (Reports)
- تقارير متعددة تشمل:
  - تقرير المبيعات
  - تقرير التحصيلات
  - تقرير أعمار الديون
  - تقرير المحفظة
  - تقرير المصنع

### 👔 تقرير المندوبين (Employee Report)
- أداء كل مندوب مبيعات
- المبيعات، التحصيلات، متوسط الفاتورة

### ❌ مديونيات معدومة (Writeoffs)
- تسجيل الديون التي لا يمكن تحصيلها
- تُخصم تلقائياً من رصيد العميل

### % نسب الشركاء (Agent Discount Rates)
- إدارة نسب الخصم التي يمنحها المصنع للوكيل
- تُستخدم لحساب تكلفة المصنع في الفواتير

### 🏢 إدارة الفروع (Branches)
- إدارة فروع الشركة المختلفة

### 🔧 الإعدادات (Settings)
- **إعدادات النظام**: اسم الشركة، العملة، الخصم الافتراضي
- **القوائم**: المحافظات، وحدات المنتجات، طرق الدفع، قائمة المحصّلين
- **Supabase**: إعداد الاتصال بقاعدة البيانات
- **المستخدمون**: إدارة المستخدمين وصلاحياتهم
- **النسخ الاحتياطي**: تصدير/استيراد بيانات النظام
- **استيراد البيانات**: استيراد من CSV/JSON

## 7. نظام الصلاحيات والأدوار

### الأدوار المدمجة:

| الدور | الاسم | الوصف |
|-------|-------|-------|
| `admin` | مدير النظام | صلاحيات كاملة على كل شيء |
| `manager` | مدير عام | صلاحيات واسعة بدون حذف |
| `sales` | مبيعات | إنشاء فواتير وتحصيلات وعملاء |
| `collector` | حسابات | تسجيل التحصيلات فقط |

### صلاحيات كل دور بالتفصيل:

**مدير النظام (admin)**:
- كل الصفحات ✅
- كل العمليات (إضافة، تعديل، حذف، تصدير) ✅

**مدير عام (manager)**:
- كل الصفحات ✅
- لا يستطيع الحذف في معظم الأقسام ❌

**مبيعات (sales)**:
- العملاء، المنتجات، الفواتير، التحصيلات، المساعدة
- لا يرى الأرباح أو التقارير المالية الكاملة ❌

**حسابات (collector)**:
- العملاء (عرض فقط)، التحصيلات، المحفظة، المساعدة

### تخصيص الصلاحيات:
يمكن لمدير النظام تخصيص صلاحيات كل مستخدم بشكل فردي من:
**الإعدادات → المستخدمون → تعديل المستخدم**

## 8. المشاكل التي تم اكتشافها وإصلاحها

### 🔴 مشكلة 1: ملف SQL يفشل عند تشغيله مرة ثانية

**المشكلة**: جدول `invoice_items` كان يفتقر إلى جملة `ON CONFLICT DO NOTHING` في نهاية الـ INSERT الخاص بالبيانات التجريبية. كان هذا يتسبب في خطأ "duplicate key value" إذا حاولت تشغيل ملف SQL للمرة الثانية.

**الإصلاح**: تم إضافة `ON CONFLICT DO NOTHING` في نهاية INSERT لجدول `invoice_items`.

---

### 🟡 مشكلة 2: تناقض صلاحيات دور "مدير عام" بين SQL والكود

**المشكلة**: الكود JavaScript يمنح دور `manager` صلاحية مشاهدة صفحة `activityLog` (سجل الحركات)، لكن ملف SQL كان لا يشمل هذا في جدول `role_permissions`.

**الإصلاح**: تم إضافة صلاحية `activityLog` لدور `manager` في ملف SQL.

---

### 🟡 مشكلة 3: أسماء جداول مرجعية غير موجودة في قاعدة البيانات

**المشكلة**: الكود يحتوي على مراجع لجداول `office_payments` و `office_incoming` في متغير `DATA_TABLES` والدوال `generateMockOfficePayments()` و `generateMockOfficeIncoming()`. هذه الجداول **لا توجد** في قاعدة البيانات لأنها كانت ميزة قديمة تم حذفها.

**التأثير**: هذه المراجع لا تسبب أي خطأ ظاهر للمستخدم. عند تسجيل الخروج، يحاول الكود حذف هذه المفاتيح من `localStorage` وهو أمر آمن ولا يسبب خطأ.

**الحل**: لا يلزم إصلاح فوري لأنه لا يؤثر على عمل التطبيق.

---

### 🟡 مشكلة 4: جدول `invoice_items` لديه عمود `created_at` لكن الكود يتجاهله

**المشكلة**: في الكود (السطر 654) يوجد تعليق يقول "invoice_items has no created_at column" وتُستثنى هذه الجدول من الترتيب بـ `created_at`. لكن في ملف SQL، الجدول **يملك** عمود `created_at` فعلاً.

**التأثير**: بنود الفواتير تُجلب من قاعدة البيانات بدون ترتيب معين. لكن هذا لا يؤثر على عمل التطبيق لأن البنود دائماً تُعرض مرتبطة بفاتورتها.

---

### 🟡 مشكلة 5: جداول `roles` و `role_permissions` موجودة في SQL لكن لا يستخدمها الكود

**المشكلة**: يُنشئ ملف SQL جدولَي `roles` و `role_permissions` ويملأهما ببيانات الصلاحيات. لكن كود JavaScript لا يقرأ من هذين الجدولين أبداً — يعتمد الكود على:
1. صلاحيات ثابتة في `DEFAULT_ROLE_PERMS` داخل الكود
2. صلاحيات مخصصة محفوظة في `app_settings.data.rolePerms` في Supabase

**التأثير**: الجدولان موجودان في قاعدة البيانات لكنهما غير مستخدمَين فعلياً. إذا عدّلت `role_permissions` في Supabase مباشرةً، لن يؤثر ذلك على التطبيق.

**الحل للمستقبل**: إذا أردت تعديل صلاحيات الأدوار، افعل ذلك من **الإعدادات → المستخدمون** داخل التطبيق نفسه.

---

### 🟡 مشكلة 6: بيانات اعتماد Supabase مُضمَّنة في الكود

**المشكلة**: أسطر 440-441 في `index.html` تحتوي على:
```
window.SUPABASE_URL  = 'https://osvtuwkrdmgqsxhntzir.supabase.co';
window.SUPABASE_ANON_KEY = 'eyJhbGciOiJIUzI1NiIs...';
```

**الإيضاح**: `ANON_KEY` (المفتاح المجهول) آمن نسبياً للنشر العام لأنه مقيَّد بسياسات RLS. لكن يُفضَّل إزالته من الكود إذا كنت ستنشر المشروع على GitHub بشكل عام، أو استخدام `.gitignore` لحماية الملف.

**تحذير**: لا تضع `SERVICE_KEY` (مفتاح الخدمة) في الكود تحت أي ظرف.

## 9. المشاكل المعروفة وحلولها

### مشكلة: لا أستطيع تسجيل الدخول

**الأسباب المحتملة والحلول:**

1. **خطأ "email not confirmed"**:
   - اذهب إلى Supabase → Authentication → Providers → Email
   - أوقف "Confirm email" (اجعلها OFF)
   - أو اذهب إلى Authentication → Users وأكّد البريد يدوياً

2. **خطأ "invalid login credentials"**:
   - تأكد أن المستخدم موجود في Authentication → Users (وليس فقط في جدول users)
   - تأكد من صحة كلمة المرور

3. **لا يوجد اتصال بـ Supabase**:
   - تحقق من SUPABASE_URL و SUPABASE_ANON_KEY في أسطر 440-441 من index.html
   - تأكد من الاتصال بالإنترنت

4. **المستخدم موقوف (Inactive)**:
   - اذهب إلى الإعدادات → المستخدمون من حساب مدير النظام
   - غيّر حالة المستخدم إلى "نشط"

---

### مشكلة: الصفحة لا تُحدَّث تلقائياً

**الحل**: اضغط زر التحديث (🔄) في أعلى الصفحة اليمنى، أو انتظر 30 ثانية للتحديث التلقائي.

---

### مشكلة: الأرقام في لوحة التحكم لا تتطابق مع توقعاتي

**الإيضاح**:
- لوحة التحكم تعرض بيانات **الشهر الحالي** افتراضياً
- لرؤية كل البيانات، اختر "الكل" في محدد الفترة الزمنية

---

### مشكلة: ظهور رسالة "جدول سجل الحركات غير موجود في Supabase"

**الحل**: تأكد أنك شغّلت ملف `database/setup.sql` بالكامل في Supabase SQL Editor. هذه الرسالة تعني أن جدول `activity_log` لم يُنشَأ.

---

### مشكلة: البيانات التجريبية لا تظهر

**الإيضاح**: ملف `setup.sql` يُضيف بيانات تجريبية فقط إذا لم تكن هناك بيانات موجودة مسبقاً (يستخدم `ON CONFLICT DO NOTHING`). إذا كانت قاعدة البيانات فارغة تماماً، ستظهر البيانات التجريبية.

---

### مشكلة: نسيت كلمة مرور مستخدم

1. اذهب إلى Supabase → Authentication → Users
2. ابحث عن المستخدم
3. اضغط على قائمة الثلاث نقاط (...) بجانب المستخدم
4. اختر "Send password recovery email" أو "Reset password"

## 11. نصائح الاستخدام اليومي

### نصيحة 1: إنشاء فاتورة جديدة

1. اذهب إلى **الفواتير**
2. اضغط **"فاتورة جديدة"**
3. اختر العميل (ابدأ بالكتابة للبحث)
4. اختر المندوب
5. اضغط **"إضافة صنف"** لكل منتج
6. ابحث عن المنتج وأدخل الكمية
7. راجع المجموع واضغط **"حفظ الفاتورة"**

### نصيحة 2: تسجيل تحصيل

1. اذهب إلى **التحصيلات**
2. اضغط **"تحصيل جديد"**
3. اختر العميل وأدخل المبلغ وطريقة الدفع
4. يمكنك ربطه برقم فاتورة محددة (اختياري)

### نصيحة 3: عرض رصيد عميل

1. اذهب إلى **العملاء**
2. ابحث عن العميل
3. الرصيد يظهر في عمود "الدين" في الجدول
4. لعرض كشف حساب تفصيلي: اضغط على زر العين (👁️) بجانب العميل

### نصيحة 4: تصدير البيانات

- من أي صفحة اضغط زر **"تصدير Excel"** لتصدير البيانات الظاهرة كـ CSV
- يمكن فتح ملفات CSV في Microsoft Excel أو Google Sheets

### نصيحة 5: النسخ الاحتياطي

من **الإعدادات → النسخ الاحتياطي**:
- اضغط **"تصدير البيانات"** لحفظ نسخة احتياطية كـ JSON
- يمكن استيراد النسخة لاحقاً من نفس الصفحة

## 12. الأسئلة الشائعة

### س: هل يعمل التطبيق بدون إنترنت؟
**ج**: لا. التطبيق يعتمد على Supabase كقاعدة بيانات سحابية وهي تتطلب اتصالاً بالإنترنت.

### س: كم عدد المستخدمين الذين يمكنهم العمل في نفس الوقت؟
**ج**: لا يوجد حد مُعيَّن. التطبيق يدعم التحديث الفوري (Realtime) بين المستخدمين.

### س: هل البيانات آمنة؟
**ج**: نعم. Supabase يستخدم سياسات Row Level Security (RLS) التي تضمن أن المستخدمين المسجَّلين فقط يمكنهم الوصول للبيانات.

### س: كيف أضيف مستخدماً جديداً؟
**ج**:
1. أضفه في **الإعدادات → المستخدمون** (داخل التطبيق)
2. ثم أضفه في **Supabase → Authentication → Users** بنفس البريد الإلكتروني وكلمة مرور

### س: كيف أعدّل دور مستخدم؟
**ج**: من **الإعدادات → المستخدمون**، اضغط تعديل بجانب المستخدم وغيّر الدور.

### س: ظهرت بيانات تجريبية في قاعدة البيانات وأريد حذفها
**ج**: من **Supabase → Table Editor**، يمكنك حذف البيانات يدوياً من كل جدول. أو من داخل التطبيق يمكنك حذف كل سجل بشكل منفرد.

### س: الفاتورة ظهرت "ملغية" بدلاً من حذفها. ما الفرق؟
**ج**: الإلغاء أفضل من الحذف لأنه يحافظ على السجل التاريخي. الفواتير الملغية لا تدخل في حسابات المبيعات أو الأرباح.

### س: ما هو "سجل الحركات"؟
**ج**: هو سجل تلقائي لجميع العمليات التي يجريها المستخدمون (إضافة، تعديل، حذف، دخول، خروج). مفيد لتتبع من فعل ماذا ومتى.

### س: كيف أغيّر شعار الشركة في الفواتير؟
**ج**: اذهب إلى **الإعدادات → إعدادات النظام** وعدّل اسم الشركة.

```

  → getRolePerms يتحقق من appState.settings.rolePerms أولاً

dbGet('table_name')         // قراءة من Supabase

dbDelete('table_name', id)  // حذف من Supabase
dbLoadSettings()            // قراءة الإعدادات
dbSaveSettings(settings)    // حفظ الإعدادات
```
